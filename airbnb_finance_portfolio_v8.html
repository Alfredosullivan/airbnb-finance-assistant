01 — Resumen

¿Qué problema resuelve?

Como anfitrión de Airbnb con múltiples propiedades, cada mes llegan pagos al banco con montos difíciles de rastrear: Airbnb descuenta comisiones, IVA, ISR y deposita días después del check-out. Cruzar manualmente el reporte de Airbnb contra el estado de cuenta bancario de BBVA era tedioso y propenso a errores.

Esta app automatiza todo: sube el CSV de Airbnb y uno o dos PDFs del banco, y en segundos obtienes cuáles pagos coinciden, cuáles están pendientes y un Excel con el reporte fiscal completo. Incluye dashboard de métricas con gráfica comparativa año a año, análisis financiero por IA con Claude, PDF ejecutivo multi-propiedad, historial con drawer lateral, autenticación JWT dual (cookie httpOnly para el browser, Bearer token para el CLI), job queue para operaciones lentas, scheduler de 3 jobs automáticos, crawler de precios de mercado (Lamudi JSON-LD) y un CLI de 7 comandos que consume toda la API.

02 — Tecnologías

Stack tecnológico

Selección deliberada de herramientas maduras y livianas. Sin frameworks pesados en el frontend para mantener control total del DOM y velocidad de carga. El CLI usa cero dependencias externas — solo módulos nativos de Node.js.

🐘

Migración completada: SQLite → PostgreSQL 18

La base de datos fue migrada de better-sqlite3 a pg con pool de conexiones. Los 14 reportes históricos fueron exportados a scripts/seed.json y reinsertados en PostgreSQL con scripts/migrate.js. Los tests usan pg-mem (PostgreSQL en memoria) para aislamiento total sin depender de una instancia real. La suite creció de 39 a 69 tests en el proceso.

🐘

¿Por qué migrar a PostgreSQL?

Mayor robustez en producción, soporte nativo para conexiones concurrentes con pool, tipos de datos más ricos (TIMESTAMPTZ, NUMERIC), y compatibilidad con Railway, Render y otros PaaS. La migración fue posible porque el código usaba SQL puro — sin ORM que impusiera restricciones.

🖥️

¿Por qué el CLI usa cero dependencias?

El CLI consume solo módulos nativos de Node.js (http, https, fs, readline, os). Sin npm install, funciona en cualquier máquina con Node. La autenticación es Bearer token en header — distinto al browser que usa cookie httpOnly.

🔐

¿Por qué JWT dual: cookie + Bearer?

Las cookies httpOnly son perfectas para el browser (inaccessibles por JS → protección XSS), pero el CLI no tiene acceso a cookies. La solución: GET /api/auth/me/token re-firma un JWT que el CLI guarda en ~/.airbnb-cli.json y envía como Authorization: Bearer.

✦

¿Por qué JSON-LD y no scraping DOM?

Lamudi usa client-side rendering — el DOM de listings llega vacío. Pero embebe los 30 listings completos en <script type="application/ld+json"> generado server-side. JSON-LD es más estable que clases CSS y no requiere Puppeteer (300MB de dependencia).

05 — Debugging

Bugs críticos resueltos

Estos son los problemas más interesantes encontrados durante el desarrollo — exactamente el tipo de situaciones que demuestran razonamiento técnico real en una entrevista.

CVE path-to-regexp@0.1.13 resuelto con Express 5 Infra

Express 4 usa path-to-regexp@0.1.13 internamente para parsear rutas. Esa versión tiene una vulnerabilidad ReDoS (Regular Expression Denial of Service) catalogada como high por GitHub Dependabot — un atacante puede enviar una URL malformada que hace que el regex tarde exponencialmente, bloqueando el event loop de Node.

Fix: npm install express@5 — path-to-regexp@0.1.13 desaparece del árbol de dependencias. 69/69 tests pasando sin cambios adicionales.

PDF BBVA: "PeriodoDEL" sin espacio Parser

El texto extraído por pdf-parse no tiene espacio entre "Periodo" y "DEL". El regex buscaba Periodo DEL y nunca matcheaba.

Fix: /PeriodoDEL\s+(\d{2}\/\d{2}\/\d{4})\s+AL\s+(\d{2}\/\d{2}\/\d{4})/

Timezone UTC vs GMT-6: depósitos descartados Parser

Al crear Date objects desde strings YYYY-MM-DD, JavaScript los interpreta en UTC. En GMT-6, "2026-02-01 UTC" son las 18:00 del 31 de enero local — el filtro de febrero los descartaba.

Fix: comparación de strings — dep.date.startsWith('2026-02')

Formato MM/DD vs DD/MM ambiguo en CSV de Airbnb Parser

El CSV usa formato americano MM/DD/YYYY. El parser asumía DD/MM, convirtiendo "02/28/2026" en "2026-28-02" — fecha inválida que luego rompía el cálculo del mes predominante.

Fix: si parte[1] > 12 entonces DD/MM, si parte[0] > 12 entonces MM/DD

DEPOSITO DE TERCERO API no detectado como abono de Airbnb Parser

En noviembre 2025, Airbnb depositó vía "DEPOSITO DE TERCERO API". El parser solo filtraba SPEI y descartaba esos depósitos.

Fix: array DESCRIPCIONES_AIRBNB con ambos prefijos, filtrar con .some()

Dashboard mostraba ingresos en 0 pese a tener reportes Backend

getDashboard() buscaba s.summary.airbnbTotal pero el campo real guardado en DB es s.summary.totalAirbnbPayouts. Los mismos campos con distintos nombres en distintos contextos — un bug de inconsistencia de schema.

Fix: fallback chain — totalAirbnbPayouts || totals.airbnbPayouts || airbnbTotal

Dashboard "vs [object Object]" en comparativa YoY Backend

El objeto de respuesta declaraba prevYear dos veces: primero como número (2025) y luego como objeto, sobrescribiendo el número. El frontend mostraba el objeto serializado.

Fix: renombrar el objeto anidado a prevYearData para evitar colisión de nombres

analysisText omitido en endpoint de polling — UI mostraba "Sin contenido" Crawler

El controller de jobs expone el resultado del polling omitiendo deliberadamente el buffer Excel para no enviar megabytes en cada tick. Al agregar market_analysis, el campo job.result.analysisText quedó también omitido. El frontend buscaba job.result?.analysis (campo que no existe) y mostraba el fallback vacío, sin error visible.

Fix: exponer analysisText: job.result?.analysisText || null en getJobStatus() + leer job.analysisText (raíz del JSON, no job.result)

markdownToHtml() cortaba el análisis en la primera tabla markdown Crawler

La implementación original usaba 5 regex secuenciales. Las líneas de tabla (| col | col |) no matcheaban ninguna regla y el parser las envolvía en <p>, rompiendo el HTML. El análisis se truncaba visualmente en el primer separador de tabla.

Fix: reescribir markdownToHtml() como parser línea a línea con estado — detecta tablas, listas, blockquotes, hr y encabezados h2/h3

Prompt de análisis de mercado confundía precios mensuales con tarifa/noche Crawler

El frontend mandaba el promedio de precios de Lamudi (7,042 MXN/mes) como currentRate. Claude lo recibía como "Tarifa actual: 7,042 MXN/noche" y generaba un análisis diciendo que la tarifa era insostenible — correcto numéricamente pero basado en datos erróneos.

Fix: eliminar currentRate del body del POST + agregar IMPORTANTE en el prompt explicando que los precios son rentas mensuales, no tarifas nocturnas

generatePriceAnalysis usaba client no definido en scope Crawler

La función usaba client.messages.create() directamente. En analysisGenerator.js, client nunca es una variable de módulo — cada función debe llamar getClient(). Resultado: ReferenceError al primer análisis de mercado.

Fix: const client = getClient() al inicio de generatePriceAnalysis()

Lamudi DOM vacío — pivot a JSON-LD server-side Crawler

Cheerio encontraba .listings__cards con 0 hijos — client-side rendering. El DOM llega vacío al servidor. Inspección del HTML estático reveló un bloque application/ld+json con los 30 listings completos generado server-side.

Fix: extraer JSON.parse() del bloque ld+json, navegar @graph[0].mainEntity["0"].itemListElement

uuid@14 ESM-only rompió Jest — revertir override Infra

Para parchear una vulnerabilidad en exceljs se agregó overrides: { "uuid": "^14.0.0" }. uuid@14 es ESM-only (export syntax), Jest corre en CommonJS y no puede parsear node_modules/uuid/dist-node/index.js. 69 tests fallando con "Unexpected token 'export'".

Fix: revertir override — la vuln es moderada, requiere pasar buf a uuid.v3/v5 (exceljs usa v4 únicamente)

ENOENT: uploads/ no existe en contenedor Railway/Docker Infra

En desarrollo, uploads/ existía porque se había creado manualmente o estava en git. En un contenedor nuevo, el filesystem arranca vacío. multer lanzaba ENOENT en el primer upload porque el directorio no existía.

Fix: fs.mkdirSync(UPLOADS_DIR, { recursive: true }) al cargar finance.routes.js, antes de diskStorage

cmdLogin verificaba /health antes de guardar la URL CLI

Dos bugs en uno: (1) apiRequest() lee config.baseUrl del archivo, pero en el primer login ese archivo no existe — la URL recién ingresada nunca se guardaba antes del request, siempre conectaba a localhost. (2) En Railway, /health puede no estar expuesto detrás del proxy.

Fix: saveConfig({ baseUrl }) sin verificar — el primer apiRequest real valida la conexión

cmdStats --month usaba query param en vez de path param CLI

El comando llamaba /api/reports?month=2026-01. El endpoint real es GET /api/reports/:month — path param, no query string. La ruta con query param no existe y devolvía 404.

Fix: apiRequest('GET', `/api/reports/${month}`) — interpolación en el path

cmdStats --year llamaba endpoint inexistente con filtro en servidor CLI

El comando llamaba /api/reports?year=2025 (no existe). El endpoint correcto es GET /api/reports/list que devuelve { reports: [...] }. El filtro por año se hace en el cliente sobre el array devuelto.

Fix: GET /api/reports/list → res.body.reports.filter(r => String(r.year) === year)

CLI: ECONNREFUSED causaba unhandled rejection y crash con stack trace CLI

El switch llamaba cmdStats(args) sin capturar el Promise rechazado. Un servidor caído mostraba el stack trace completo de Node al usuario en lugar de un mensaje amigable.

Fix: run() wrapper central — .catch() unificado con mensajes por código de error (ECONNREFUSED, ENOTFOUND)
06 — Prep para entrevista

Preguntas y respuestas

Respuestas concretas para las preguntas más comunes en entrevistas Full Stack Junior, basadas exactamente en lo que se construyó.

Cuéntame de un proyecto personal que hayas construido
Construí una app web full stack para automatizar la reconciliación financiera de propiedades en Airbnb. La app lee el CSV de Airbnb y los PDFs del banco BBVA, los cruza por monto y mes, y genera un Excel con el reporte fiscal completo. Incluye dashboard con métricas KPI, análisis por IA con Claude, crawler de precios de mercado, job queue asíncrono, scheduler automático y un CLI de 7 comandos que consume toda la API. El resultado: 100% de match rate en la prueba real con datos reales de febrero 2026.
¿Cómo diseñaste la arquitectura y por qué?
Usé el patrón Routes → Controllers → Services → Utils en Node.js con Express. Para operaciones lentas (Claude API, crawl de mercado) agregué un MemoryQueue + Worker desacoplado del request HTTP: el controller encola el job y responde 202 inmediatamente, el worker lo procesa en background, el cliente hace polling con GET /api/jobs/:id.

Esta separación permitió migrar la BD de SQLite a PostgreSQL sin tocar ningún service — solo se reemplazó el cliente en src/database/client.js. También facilitó agregar el crawler y el CLI sin romper ningún módulo existente.
¿Cómo implementaste la autenticación para el CLI cuando el token está en una cookie httpOnly?
Las cookies httpOnly son inaccessibles desde JavaScript — ni siquiera Node.js puede leerlas. El CLI no puede usar el flujo de cookie del browser. La solución fue agregar el endpoint GET /api/auth/me/token que re-firma un nuevo JWT con el mismo payload { userId, username } usando jwt.sign() y lo devuelve en el body JSON.

El middleware de autenticación soporta dual auth: primero busca req.cookies.token (browser), y si no existe lee Authorization: Bearer <token> (CLI). Cookie toma precedencia porque es más segura (httpOnly). El CLI guarda su token en ~/.airbnb-cli.json — fuera del directorio del proyecto para no committearse accidentalmente a git.
¿Cómo funciona el sistema de jobs en background?
Implementé un MemoryQueue basado en un Map de JavaScript. Cada job tiene un ID único (uuid), un tipo (excel_generation o market_analysis) y datos para procesarlo. El analysisWorker corre un loop de polling con setTimeout cada 2 segundos — no setInterval porque si el job tarda 30 segundos y el intervalo es 2 segundos, setInterval acumularía 15 ciclos en paralelo. Con setTimeout, el siguiente ciclo empieza solo cuando el actual termina.

El resultado (Excel como base64 o análisis de mercado como texto) se guarda en job.result. El endpoint GET /api/jobs/:id/download convierte el base64 de vuelta a Buffer y lo sirve con el header Content-Disposition: attachment.
¿Cómo manejaste el caso de Lamudi con client-side rendering?
Guardé el HTML estático en disco e inspeccioné con cheerio. El selector .listings__cards existía pero con 0 hijos — señal clásica de CSR: el HTML llega vacío y JavaScript inyecta el contenido en el browser. Sin ejecutar JavaScript no hay listings.

La solución fue buscar datos server-side en el HTML. Encontré un bloque <script type="application/ld+json"> de 67KB con los 30 listings completos como schema.org/SingleFamilyResidence. Parsear JSON es más estable que selectores CSS y no requiere Puppeteer (300MB de dependencia). El precio no tiene campo propio — está en el texto de description — así que uso 5 patrones de regex en orden de especificidad para extraerlo.
¿Cómo integraste la API de Anthropic y cómo controlas los costos?
Uso el SDK oficial @anthropic-ai/sdk. Para el análisis mensual, el prompt incluye los datos reales del reporte (ingresos, noches, match rate, transacciones). Limité max_tokens a 1500 para respuestas concisas.

Para controlar costos implementé caché en DB: el primer análisis llama a la API y guarda el resultado en el JSON del reporte. Los accesos siguientes leen de la DB sin costo adicional. Para el análisis de mercado, la función generatePriceAnalysis() recibe el prompt ya construido con los datos del crawl — la misma API pero desacoplada del dominio financiero.
¿Cómo resolviste el problema de los timezones?
Bug sutil: new Date('2026-02-01') se interpreta como medianoche UTC. En GMT-6 (Mérida), eso son las 18:00 del 31 de enero. Entonces un depósito del 1 de febrero quedaba "fuera" del rango de febrero.

La solución fue no usar Date objects para comparar fechas. Todas las fechas están normalizadas a strings YYYY-MM-DD y se comparan directamente: dep.date.startsWith('2026-02'). Los strings YYYY-MM-DD son ordenables lexicográficamente y son timezone-agnostic.
¿Cómo implementaste el análisis de mercado con IA en el frontend?
El backend ya tenía el endpoint POST /api/crawler/analyze que encola el análisis con Claude y devuelve un jobId con HTTP 202. El reto era conectarlo a la UI de forma que no bloqueara la pantalla mientras Claude piensa.

Implementé el patrón de polling asíncrono: el frontend hace POST, recibe el jobId, y luego hace GET /api/jobs/:jobId cada 3 segundos con un máximo de 20 intentos (60 segundos de timeout). Cuando el status es completed, renderiza el analysisText pasándolo por un parser de markdown manual que convierte tablas, listas, blockquotes y encabezados a HTML sin dependencias externas.

Encontré dos bugs durante la integración: el endpoint de polling omitía analysisText para evitar enviar buffers de Excel pesados, y el campo estaba en la raíz del JSON (job.analysisText) no en job.result.analysisText. Ambos invisibles sin leer el controller.
¿Qué harías diferente si lo reharías?
Tres cosas: primero, TypeScript desde el inicio — los bugs de formato de fecha (MM/DD vs DD/MM, totalAirbnbPayouts vs airbnbTotal) habrían sido detectados en compile time. Segundo, Express 5 desde el inicio — Express 4 tiene una vulnerabilidad de ReDoS en path-to-regexp@0.1.13 que solo se resuelve migrando a Express 5 (que usa path-to-regexp@^8.0.0). Tercero, un store de sesión por usuario en lugar del store global compartido — importante cuando escale a múltiples usuarios simultáneos.
💡 Puntos clave para la entrevista

(1) Parsear documentos reales de instituciones mexicanas (BBVA, Airbnb) — no datos simulados. (2) Autenticación dual cookie+Bearer: el patrón detrás y por qué no se puede usar la misma cookie en el CLI. (3) Job Queue con MemoryQueue: por qué setTimeout y no setInterval, cómo el worker despacha por tipo. (4) Crawler: detectar CSR vs SSR, pivotar a JSON-LD sin Puppeteer, extraer precio de texto libre con regex. (5) CLI zero-dependency: por qué, qué módulos nativos usa, cómo maneja errores de red con un wrapper central. (6) Resolver bugs no triviales: timezone, ambigüedad MM/DD, ESM vs CJS en Jest, colisión de nombres en JSON. (7) Mantener 69 tests pasando a través de múltiples refactors incluyendo cambios de schema y nuevos módulos. (8) Patrón de polling asíncrono: POST 202 + jobId, GET polling cada 3s con timeout de 60s — por qué 202 y no 200, y qué problema resuelve frente a mantener el request abierto.

07 — Roadmap

Próximamente

Mejoras planificadas que llevarán el proyecto de herramienta personal a aplicación lista para producción. Cada cambio tiene una justificación técnica clara — el tipo de decisiones que se discuten en entrevistas senior.

✅ UI análisis IA de mercado Completado

Botón "Analizar con IA" integrado en la sección del crawler. Implementa el patrón de polling asíncrono completo (POST 202 → GET polling → render), parser markdown manual con soporte de tablas y blockquotes, y prompt corregido que distingue rentas mensuales de tarifas vacaciones.

  • analyzeMarketUI() con polling cada 3s, timeout 60s
  • markdownToHtml() reescrito como parser línea a línea
  • jobs.controller expone analysisText sin buffer Excel
  • max_tokens 1000 → 2000 en generatePriceAnalysis

✅ Migración a Express 5 Completado

Express 4.x usa path-to-regexp@0.1.13 que tiene una vulnerabilidad de ReDoS (CVE GHSA-9wv6-86v2-598j) catalogada como high por GitHub Dependabot. La única solución es migrar a Express 5, que delega el routing a router@^2.x con path-to-regexp@^8.0.0.

  • npm install express@5 — actualizar package.json
  • Revisar breaking changes en async error handling
  • Verificar que los 69 tests siguen pasando
  • Eliminar la alerta de Dependabot definitivamente
  • 69/69 tests pasando sin modificar una sola línea de código

✅ Migración a TypeScript Completado

Los bugs más costosos del proyecto (campos mal nombrados, formatos de fecha, colisiones en JSON) habrían sido errores de compilación con TypeScript. La migración es incremental — un archivo a la vez, empezando por los services.

  • tsconfig.json con allowJs:true — migración incremental sin romper .js existentes
  • src/types.ts — 10 interfaces del dominio (Reservation, AirbnbPayout, BankDeposit, etc.)
  • csvParser.ts — fix Date.getTime(), tipos en todas las funciones
  • pdfParser.ts — BBVAMeta local, stub Reservation completado
  • comparator.ts — type narrowing de BankDataInput, 3× Date.getTime()
  • ts-jest configurado — Jest compila .ts con TypeScript directamente
  • 69/69 tests pasando en cada paso de la migración

El Dockerfile y docker-compose están listos. El siguiente paso es conectar el repositorio a Railway para deploy automático en cada push a main, con PostgreSQL gestionado por la plataforma.

El diseño Open/Closed de TARGETS en crawlerService.js permite agregar fuentes sin modificar el orchestrator. Trovit respondió OK (36K chars) pero aún no fue inspeccionado para encontrar la estructura de listings.

🔑 OAuth / SSO con Google Próximo

El flujo actual de registro y login con email/password funciona, pero para un producto multi-usuario real, OAuth reduce la fricción de onboarding y elimina la responsabilidad de manejar contraseñas.

  • Passport.js con estrategia Google OAuth 2.0
  • Columna google_id en tabla users (nullable)
  • Compatibilidad con cuentas existentes via email
  • Botón "Continuar con Google" en los modales de auth

📱 PWA + notificaciones Próximo

Convertir la app en Progressive Web App permite instalarla en el homescreen y recibir notificaciones push cuando un reporte mensual está listo o hay discrepancias importantes.

  • manifest.json con íconos del sistema de diseño
  • Service Worker para cache offline de la UI
  • Push notifications via Web Push API
  • Alerta automática si match rate cae debajo de 80%
💡 Para la entrevista

El roadmap muestra madurez de criterio técnico: Express 5 antes de TypeScript porque el CVE de seguridad tiene impacto real en producción. TypeScript antes de OAuth porque los tipos previenen bugs que OAuth no toca. El crawler de Trovit antes de PWA porque agrega valor de datos directo. El orden importa — y poder explicarlo es lo que diferencia a un desarrollador que entiende su stack.
